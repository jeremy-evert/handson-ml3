## Your Mission

Build `tools/notebook_enricher/enrich_runner.py` — a sequential three-stage enrichment runner

## The Three-Stage Model

Treatment is applied in three separate waves across all notebooks.
Each stage is a distinct `codex exec` job per notebook.

### Stage 1 — Chapter Intros
For each numbered notebook, in order (01 → 19):
- Check: is there a substantive chapter-level intro markdown cell before the Setup section?
  - Substantive = more than 200 words, contains learning objectives and ML context
  - A thin heading or one-sentence cell does NOT count
- If NO: generate a full chapter intro and insert it at the correct position
- If YES: skip this notebook (log the skip)
- Write a V1-style execution record for this notebook+stage to `notes/enrichment/`
- Move to the next notebook

**NB06 is already done — skip it in all stages.**

---

### Stage 2 — Goal Before Each Code Cell
For each numbered notebook (except NB06), in order:
- For each non-empty code cell in the notebook:
  - Check: is there a markdown cell immediately before this code cell?
    - If yes, does it use canonical vocabulary (`### Goal Before This Cell`)?
    - If yes but legacy vocabulary (`**Why run this cell**:`, `**Goal:**`, `**What it is**:`): treat as LEGACY — replace with canonical format
    - If no preceding markdown cell at all: treat as MISSING — insert one
    - If canonical and substantive (>3 sentences): treat as PRESENT — skip
  - For MISSING and LEGACY cells: generate a new goal markdown cell using the treatment spec
- Write a V1-style execution record for this notebook+stage to `notes/enrichment/`
- Move to the next notebook

---

### Stage 3 — Implementation Notes After Each Code Cell
For each numbered notebook (except NB06), in order:
- For each non-empty code cell in the notebook:
  - Check: is there a markdown cell immediately after this code cell?
    - Canonical: `### Implementation Notes After This Cell`
    - Legacy: `**Result**:`, `**Why this matters**:`
    - Rules same as Stage 2: MISSING → insert, LEGACY → replace, PRESENT → skip
  - For MISSING and LEGACY cells: generate a new implementation notes cell
- Write a V1-style execution record for this notebook+stage to `notes/enrichment/`
- Move to the next notebook

## How the Runner Works

The runner does NOT generate the markdown content itself via Python string templates.
It generates a **focused prompt file** for each notebook+stage, then calls `codex exec`
to do the writing work — the same way `run_prompt.py` works.

### Per-job prompt generation

For each (notebook, stage) pair, the runner generates a temporary prompt file at:
`notes/enrichment/active_prompt.md`

That prompt file contains:
1. The notebook path
2. The stage being run (1, 2, or 3)
3. The treatment spec (copied from `prompts/001_assess_and_finish_05_06_07.md`)
4. The current cell inventory for this notebook (generated by scanning the notebook JSON)
5. Specific instructions for this stage only
6. The safeguard instructions (see below)
7. The output contract (what file to modify, what record to write)

### Codex exec call

```python
import subprocess
result = subprocess.run(
    ["codex", "exec", "--output-last-message", output_file, "-"],
    input=prompt_content,
    capture_output=True,
    text=True,
    cwd=repo_root
)
```

Same pattern as `run_prompt.py`. Capture stdout, stderr, return code.

### V1-style execution record

After each codex exec call, write a record to `notes/enrichment/{notebook_stem}_stage{N}__{timestamp}.md`.

Use the same V1 section structure from `tools/codex/V1_Execution_Record_Artifact.md`:
- Section 1: Identity (run_id, prompt_file, prompt_stem, started_at_utc, finished_at_utc)
- Section 2: Execution Facts (exit_code, execution_status, execution_summary)
- Section 3: Review Facts (review_status: UNREVIEWED, review_summary, reviewed_by, reviewed_at_utc)
- Section 4: Failure Analysis (failure_type, failure_symptom, etc.) — blank if succeeded
- Section 5: Resource/Cost Facts (duration_seconds, cells_scanned, cells_added, cells_replaced)
- Section 6: Prompt Text (the generated prompt for this job)
- Section 7: Codex Final Output (the last message from codex exec)
- Section 8: Stderr (if any)

Import `v1_record_validation.py` from `tools/codex/` to validate the record before writing.

### Safety: notebook integrity check

Before writing any modified notebook back to disk:

1. Read the original notebook's code cells into memory BEFORE calling codex exec
2. After codex exec completes, read the modified notebook
3. Compare every code cell source byte-for-byte
4. If any code cell source differs → abort, log the mismatch, mark the record FAILED, do NOT overwrite
5. Write to `{notebook}.tmp` first, validate JSON round-trips, then rename to final path

This check is a hard stop. Log clearly which cell index was corrupted and what changed.

## notebook_scanner.py — What It Must Detect

For each cell in a notebook, classify it as:

```python
@dataclass
class CellState:
    index: int
    cell_type: str           # 'code' or 'markdown'
    is_empty: bool           # True if source is whitespace only
    has_goal_before: str     # 'canonical', 'legacy', 'thin', 'missing'
    has_impl_after: str      # 'canonical', 'legacy', 'thin', 'missing'
    goal_cell_index: int     # index of the preceding markdown cell, or -1
    impl_cell_index: int     # index of the following markdown cell, or -1
```

Canonical detection strings:
- Goal: `### Goal Before This Cell`
- Impl: `### Implementation Notes After This Cell`

Legacy detection strings (from NB07 analysis):
- Goal: `**Why run this cell**`, `**Goal:**`, `**What it is**`, `**Principle:**`
- Impl: `**Result**:`, `**Why this matters**`, any markdown cell immediately following
  a code cell that starts with bold text and is < 100 words

Chapter intro detection:
- Position: one of cells 0–5
- Length: > 200 words = substantive, 50–200 words = thin, < 50 words = heading only
- Must appear before a cell whose source starts with `# Setup` or `# setup`

## prompt_builder.py — What the Generated Prompt Must Include

Each generated prompt passed to `codex exec` must contain:

1. **The notebook path** and which stage is being run
2. **The treatment spec** — the full cell format from prompt 001:
   - Chapter intro: 300-500 words, 5 learning objectives, ML landscape context, vocabulary
   - Goal cell: 4-8 sentences, ⚙️ Plumbing / ✨ Better practice markers
   - Implementation cell: 3-6 sentences, implementation choices, what to look for in output
3. **The cell inventory** — a list of which cells need treatment (from notebook_scanner.py),
   so codex knows exactly which cells to target without re-scanning
4. **The gold standard** — a 2-3 cell excerpt from NB06 showing canonical format in practice
5. **Hard constraints** — do not modify any code cell; only add or replace markdown cells;
   write to `{notebook}.tmp` not the original; verify code cells before finalizing

The prompt must be specific enough that codex does not need to make judgment calls
about format, placement, or scope. The scanner made those decisions; the prompt delivers them.

## Completion Criteria

The runner is done when:
- All three stages can run end-to-end on a single test notebook (`--notebook 07 --stage 1`)
- The notebook integrity check correctly aborts if a code cell is touched (write a test case)
- V1 execution records are written to `notes/enrichment/` and pass `v1_record_validation.py`
- `--dry-run` mode prints the cell inventory without writing anything
- README documents all usage patterns

**Do not run the full pipeline on all 19 notebooks.** Build and verify the runner only.
A single test run on NB07 Stage 1 is the acceptance test.
